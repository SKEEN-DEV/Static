Overview

We need to enhance the jesEnquiry/findByContact endpoint to support different data requirements for widget vs. full tab views, while maintaining security for non-subscriber users.

Frontend Status: The frontend code is already updated and ready to consume this API. The JesEnquiryService.FindByContact() method now accepts the optional parameters, and the components are prepared to handle the categorySummary array when it's returned for non-subscribers.

API Endpoint

GET /api/jesEnquiry/findByContact

Query Parameters

Parameter Type Required Default Description
id int ✓ Yes - Contact ID to fetch enquiries for
includeEnquiries bool No false Frontend request to include enquiry details (subject to backend permission check)
includeOpen bool? No true Include open enquiries in results (only applied if user has access)
includeClosed bool? No true Include closed enquiries in results (only applied if user has access)
limit int? No null Maximum number of enquiries to return (for widget use)

Security Model

Security Architecture: The backend validates user permissions from the auth token and applies business logic based on request parameters. The frontend indicates intent via includeEnquiries, but the backend enforces who can actually access enquiry data.

Security Requirements

  1. Extract authenticated user from auth token/session
  2. Query database to check if user has active JES subscriptions (permission validation)
  3. Apply business logic: shouldReturnEnquiries = includeEnquiries && userHasSubscriptions
  4. If not authorized: Return empty result array + aggregate categorySummary
  5. If authorized: Return filtered enquiry results based on includeOpen/includeClosed/limit
  6. Never expose individual enquiry details to non-subscribers
Flexibility: This approach allows the frontend to explicitly request enquiries or summary-only data, while the backend maintains security enforcement. Future business logic changes (e.g., trial access, limited previews) can be implemented without changing the API contract.
Category Counts Are Safe: The categorySummary with counts is safe to expose because it only contains aggregate data (e.g., "Beauty appeared 12 times"). It doesn't reveal enquiry summaries, deadlines, contact details, or any other sensitive information from individual enquiries.

Backend Requirements

1. New DTO Class: CategorySummaryItem

Required Properties:
  • type (string): "consumer" or "professional"
  • value (int): Bitwise flag value (e.g., 2, 4, 8, 16...)
  • name (string): Category name (e.g., "Beauty", "Technology")
  • count (int): Frequency count - how many enquiries have this category

2. Update JesEnquiryApiResponseDto

Add New Property:
  • categorySummary (List<CategorySummaryItem>): Array of category frequency data
  • Return this for non-subscribers with empty result array
  • Return null for subscribers (they get actual enquiry results)

3. Controller Method Changes

Critical Security Requirements:
  1. Extract authenticated user from auth token/session (never trust frontend)
  2. Check if user has active JES subscriptions via database lookup
  3. If no subscriptions: Return empty result + categorySummary + statistics
  4. If has subscriptions: Return filtered enquiries based on parameters
Query Parameter Handling:
  • includeOpen + includeClosed: Filter enquiries by open/closed status
  • limit: Restrict number of results (used for widget - limit=5)
  • Apply filters only for subscribers

4. Category Summary Aggregation Logic

Required Algorithm (for non-subscribers only):
  1. Load all enquiries for the contact (unfiltered)
  2. Extract individual category flags from bitwise values:
    • For each enquiry, check CatConsumer and CatProfessional fields
    • Use bitwise AND to test each category flag: (enquiry.CatConsumer & categoryValue) != 0
  3. Count frequency: Track how many times each category appears across all enquiries
  4. Sort by count descending: Most frequent categories first
  5. Take top 10-15 categories
  6. Map to CategorySummaryItem: Include type, value, name, count
Example Category Extraction:

If an enquiry has CatConsumer = 6 (bitwise), that means flags 2 (Beauty) and 4 (Fashion) are set. Both should be counted separately in the frequency map.

5. Response Patterns

User Type result categorySummary statistics user
Non-subscriber [] (empty array) Array with frequency data (sorted) Stats object null
Subscriber (widget) 5 closed enquiries null Stats object Subscription flags
Subscriber (tab) All open + closed null Stats object Subscription flags

Frontend Implementation (TypeScript/React)

1. Service Layer - API Call with Parameters (✅ Complete)

// JesEnquiryService.ts
async FindByContact(
    id: number, 
    options?: {
        includeEnquiries?: boolean;
        includeOpen?: boolean;
        includeClosed?: boolean;
        limit?: number;
    }
) {
    const result = await getApi<AIMC.Baltic.Dto.JesEnquiry.JesEnquiryApiResponseDto>({
        endPoint: 'jesEnquiry/findByContact',
        additionalParams: { 
            id: id,
            includeEnquiries: options?.includeEnquiries,
            includeOpen: options?.includeOpen,
            includeClosed: options?.includeClosed,
            limit: options?.limit
        } 
    });
    return result;
}

}

Category Display Flow

  1. Backend sends: categorySummary array sorted by frequency (most common first)
  2. Frontend receives: Pre-sorted array with frequency counts
  3. Frontend takes: First 4 items (no sorting needed - backend already sorted)
  4. Frontend maps: Summary items to full CategoryInfo objects using config lookups
  5. Component renders: Top 4 most relevant categories in promo box

Why This Design Works

  • Backend does sorting: More efficient - database-level aggregation and sorting
  • Frontend just maps: Simple transformation from summary to display format
  • No client-side sorting: Categories arrive pre-sorted by popularity
  • Config-based display: Uses existing JesEnquiryConsumerCategories/ProfessionalCategories for consistency
  • Same component: JesEnquiryCategories component works for both subscribers and non-subscribers
