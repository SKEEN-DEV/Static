Overview

The Contact Suggestions Modal is a feature within the MCD Portal that helps users manage contact information suggestions from their living list. This document details the improvements made to the modal, including immediate save actions, error handling, multiple view modes, and performance optimizations.

Project Status: Testing 🧪 Planned features are ready for testing.

Key Features Implemented

Immediate Save Actions

Actions save immediately upon click with a 3-second undo window, replacing the previous checkbox selection pattern.

Multiple View Modes

Three distinct views (Grid, List, Compact/Table) with responsive behavior and persisted user preferences.

Error Handling

Inline error overlays on failed cards with retry functionality and auto-clear after 5 seconds.

Smooth Animations

Fade-in and fade-out animations for all overlay states, synchronized with state timeouts for polished UX.

Performance Optimization

Strategic memoization helps prevent unnecessary re-renders when working with large datasets.

Safety Features

Close button disabled during processing, duplicate action guards, and proper timeout cleanup.

Technical Architecture

Frontend Stack

  • Framework: React with TypeScript
  • Dependency Injection: Inversify
  • Styling: SCSS modules with Lumina design tokens
  • Design System: Custom Lumina UI components
  • Hooks: useState, useEffect, useMemo, useCallback, useRef

UX Workflow Improvements - Implementing Immediate Accept/Decline Actions

The modal replaces the previous checkbox selection pattern with immediate save actions, providing a more intuitive and responsive user experience with a safety net for accidental clicks.

1. User Action Initiated

User clicks Accept or Dismiss button on a contact suggestion card.

2. Pending State (3 seconds)

Blue overlay appears with "Undo" button. Action can be cancelled during this window.

3. Processing

API call to livingListService.submitActions() executes after timeout expires.

4a. Success Path

Green success overlay appears for 1.5 seconds, then card is removed from the list.

4b. Error Path

Red error overlay appears for 5 seconds with retry option, card remains in list.

Component Architecture

  • LuminaActionOverlay: Unified component for pending/success/error states
  • LuminaContactList: Accepts renderActions callback for custom action buttons
  • LuminaViewControl + LuminaViewToggleButton: View mode switching interface
  • useKeyValuePreferences: Centralized preference management hook

useKeyValuePreferences Hook

Purpose: Centralized hook for managing user preferences with backend persistence.

Hook Interface

const {
    saveSuggestionsView,
    loadSuggestionsView,
    saveExpandedView,
    loadExpandedView
} = useKeyValuePreferences();

Implementation Details

// Hook implementation
export const useKeyValuePreferences = () => {
    const keyValueService = useInjection(IKeyValueStore);
    
    const SUGGESTIONS_VIEW = 'lumina-suggestions-view';
    const EXPANDED_VIEW = 'lumina-expanded-view';
    
    const saveSuggestionsView = async (view: 'grid' | 'list' | 'table') => {
        await keyValueService.set(SUGGESTIONS_VIEW, view);
    };
    
    const loadSuggestionsView = async (): Promise<string | null> => {
        return await keyValueService.get(SUGGESTIONS_VIEW);
    };
    
    return {
        saveSuggestionsView,
        loadSuggestionsView,
        saveExpandedView,
        loadExpandedView
    };
};

Benefits of Centralization

  • Single Source of Truth: All preference keys defined in one place
  • Type Safety: TypeScript ensures correct value types
  • Reusability: Multiple components can use the same preferences
  • Backend Integration: Automatic persistence to KeyValueStore service
  • Error Handling: Centralized error handling for preference operations

Load Once Pattern (Critical for Performance)

// Load saved preference once on mount - prevents flickering
useEffect(() => {
    const savedView = await loadSuggestionsView();
    if (savedView) setViewMode(savedView);
    // eslint-disable-next-line react-hooks/exhaustive-deps
}, []); // Empty dependency array = runs once only

// Save on every view change (async, fire-and-forget)
onClick={() => {
    setViewMode('grid');
    saveSuggestionsView('grid');
}}
Performance Note: The empty dependency array is critical. Including loadSuggestionsView in the dependencies causes re-runs during view changes, creating a render loop that manifests as flickering.

catch (error) {

        const next = new Set(prev);
        next.delete(suggestionId);

    });

        setFailedActions(prev => {
            const next = new Map(prev);
            next.delete(suggestionId);
            return next;
        });
    }, 5000);
}

Technical Challenges & Solutions

Throughout development, several technical challenges emerged that required iterative problem-solving. This section documents the journey from problem discovery to final solution.

Challenge 1: View Mode Flickering

The Problem: Screen would flash rapidly when switching between view modes (Grid, List, Table).

Issue Discovered

I found issues with flickering when switching between view modes. Screen would flash rapidly during transitions.

Fix Attempt 1: Memoize Overlay Settings

Hypothesis: overlaySettings() function created new object references on every render.
Solution: Used useMemo with viewMode dependency.
Result: ❌ Still flickering.

Fix Attempt 2: Memoize Contacts Array

Hypothesis: rows.map() created new array reference on every render.
Solution: Used useMemo with rows dependency.
Result: ❌ Still flickering.

Fix Attempt 3: Fix useEffect Dependencies

Root Cause: loadSuggestionsView in dependency array caused re-runs during view changes.
Solution: Empty dependency array with eslint-disable comment for load-once pattern.
Result: ✅ Flickering resolved!

Key Insight: The issue wasn't about memoization—it was about the useEffect dependency array causing the view preference to reload every time it changed, creating a feedback loop.

Challenge 2: Error Handling Design Evolution

The Problem: Users couldn't identify which contact failed when actions failed, and temporary toasts disappeared too quickly.

Version 1: Generic Toast Notifications

Implementation: Showed a generic toast notification on error.
Problem: Disappeared quickly, users couldn't identify which contact failed.
Result: ❌ Poor user experience.

Version 2: Custom Error JSX

Implementation: Added custom error overlay on cards with emoji warning icon.
Improvement: Better visibility, persistent on card.
Problem: Inconsistent styling, not aligned with Lumina design system.
Result: ⚠️ Functional but not polished.

Version 3: Unified Component (Final Solution)

Implementation: Extended LuminaActionOverlay to support error type with red X icon.
Benefits: Consistent with success/pending overlays, professional styling, reusable component.
Result: ✅ Polished, consistent user experience.

Key Insight: Extending an existing component rather than creating custom JSX ensured consistency and maintainability while solving the UX problem.

Challenge 3: Timeout Management Complexity

The Problem: Multiple overlapping timeouts for pending, success, and error states with different durations required careful orchestration.

Initial Approach: Individual Timeouts

Implementation: Set timeouts inline within action handlers.
Problem: Memory leaks when component unmounted, inconsistent cleanup, hard to track.
Result: ❌ Unreliable and caused performance issues.

Improved Approach: Centralized Tracking

Implementation: Used refs to store timeout IDs, cleanup in useEffect return functions.
Improvement: Proper cleanup on unmount.
Problem: Still complex to coordinate 3 different timeout durations (3s, 1.5s, 5s).
Result: ⚠️ Better but still fragile.

Final Solution: Map-Based State + Ref Tracking

Implementation: Separate Maps for each state (pending/success/failed) + comprehensive cleanup useEffect.
Benefits: Clear state transitions, automatic cleanup, easy to debug.
Result: ✅ Robust timeout management with proper cleanup. 

// Comprehensive cleanup on unmount
useEffect(() => {
    return () => {
        // Clear all pending timeouts
        if (undoTimeoutRef.current) clearTimeout(undoTimeoutRef.current);
        if (successTimeoutRef.current) clearTimeout(successTimeoutRef.current);
        if (errorTimeoutRef.current) clearTimeout(errorTimeoutRef.current);
    };
}, []);
Key Insight: Using Map data structures for state and refs for timeout IDs created a clean separation of concerns, making the code maintainable and preventing memory leaks.

Lessons Learned

  • Debug Systematically: Isolate variables one at a time when troubleshooting (memoization → dependencies → side effects)
  • Extend Don't Rebuild: Reuse existing components when possible rather than creating one-off solutions
  • State Architecture Matters: Well-structured state (Maps over arrays) makes complex interactions easier to manage
  • Cleanup is Critical: Always implement proper cleanup for timeouts, subscriptions, and side effects
  • Iterate Towards Quality: First make it work, then make it consistent, then make it polished

        return { hasMissingFields, missingFieldsCount, metadata };
    });
}, [rows]); // Only recompute when rows change, not on every render
Why contactMetadata needs useMemo:
  • Expensive computation (iteration, regex parsing, JSON operations)
  • Referential stability required for React prop comparison
  • Frequent re-renders from pendingActions, successActions, failedActions, viewMode changes
  • Only needs recalculation when rows change (load, removal, bulk actions)

Performance Best Practices Applied

  • Use useMemo for values that are expensive to compute or need referential stability
  • Use useCallback for functions passed as props
  • Empty dependency arrays for load-once patterns (be explicit with eslint-disable)
  • Map/Set data structures for O(1) operations instead of arrays
  • Refs for mutable values that don't trigger re-renders (timeout handles)

Lumina Gradient Logos with SVG Masks

As part of the broader Lumina UI design system, we developed a gradient logo system using SVG masks and gradient images. This technique creates visually striking, brand-consistent logos with smooth color transitions.

Technical Implementation

Core Concept: SVG masks applied to gradient images allow complex shapes to be "cut out" of colorful gradients, creating logos that maintain brand identity while offering visual richness.

Component Architecture

// LuminaGradientLogo.tsx
export enum LuminaGradientLogoVariant {
    Full = 'full',  // Complete logo with text and star
    Star = 'star'   // Star icon only
}

interface LuminaGradientLogoProps {
    variant: LuminaGradientLogoVariant;
    width?: number | string;
    height?: number | string;
    className?: string;
    onClick?: () => void;
    role?: string;
    tabIndex?: number;
    onKeyDown?: (e: React.KeyboardEvent<SVGSVGElement>) => void;
}

SVG Mask Technique

The implementation uses SVG <mask> elements to define the logo shape, then applies a gradient image through the mask:

<svg width="1201" height="219" viewBox="0 0 1201 219">
    <defs>
        <!-- Define the mask shape -->
        <mask id="luminaMask" maskUnits="userSpaceOnUse">
            <path
                d="M205.237593,13.6759835 C189.042445..."
                fill="white"
            />
        </mask>
    </defs>

    <!-- Apply gradient image through mask -->
    <image
        href={GradientImage}
        width="1201"
        height="219"
        preserveAspectRatio="xMidYMid slice"
        mask="url(#luminaMask)"
    />
</svg>

Logo Variants

Full Logo

  • Dimensions: 1201×219px
  • Contains: Complete "Lumina" text + star icon
  • Use Case: Headers, splash screens, branding
  • ViewBox: 0 0 1201 219

Star Icon

  • Dimensions: 212×213px
  • Contains: Star icon only
  • Use Case: App icons, small spaces, menu headers
  • ViewBox: 0 0 212 213

Gradient Asset

The gradient is stored as a PNG image (Gradient.png) with:

  • Color Transitions: Smooth blend from Lumina blue to complementary accent colors
  • Aspect Ratio: Wide format (1201×219) for full coverage
  • Slicing: preserveAspectRatio="xMidYMid slice" ensures no distortion
  • Reusability: Single gradient image used for all variants via different masks

Key Features

  • Scalable: SVG format scales to any size without quality loss
  • Accessible: Supports ARIA roles, tabIndex, and keyboard navigation
  • Interactive: Optional onClick handlers with cursor styling
  • Flexible Sizing: Width/height props allow custom dimensions
  • Brand Consistent: Unified gradient across all logo variants

  • Action Overlays: Adjust size and padding based on view mode and screen size
